Reading Roundup: what's in a type?

The Ethical Compiler: about the fallacy of deriving “what ought to be” from “what is”, and the importance of expressing/capturing intention in the language. The first time I’ve seen a toot quoted in a paper:

There must be compiler developers whom the language standard protects but does not bind, alongside developers whom the language standard binds but does not protect.

(A small parallel to this paper: the Go style guide recommends expressing test assertions as got compared to want, by those names; i.e. an explicit “is” and “ought to be”.)

Related work, mostly in the citations of the above:

The Correctness-Security Gap, calling out that the (C/C++) abstract machine does not capture important security properties (side channels, uncleared memory). The source code may express an intention that is not captured in the formal semantics of the language - it “does not bind” the compiler’s behavior.

Ghosts of Departed Proofs (public PDF here), on types that capture additional properties of the value. Rust’s NonZeroUsize is a nice recent example. Another take on the same idea: Parse, don’t validate.

The theory underlying these is The Curry-Howard Correspondence: a type is a proposition, and an instance of the type is a proof of the proposition. I think “Ghosts” and “Parse” show the useful aspects, but don’t require diving in to the underlying theory; it can be argued that the theory side isn’t very useful in engineering contexts.
