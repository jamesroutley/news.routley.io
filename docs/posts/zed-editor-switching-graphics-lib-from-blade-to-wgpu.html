sometimes it has long-time problems without fixes

Sure, wgpu has issues too. The difference is that it is actively maintained and a core dependency of many big projects. There is generally a big interest in getting significant things fixed. Blade on the other hand has open PRs (not even just issues) with small fixes that have a big impact (breaking issues) for downstream apps that haven't been merged in half a year. I don't want to attack the author, I know maintaining OSS is a lot of work and usually doesn't pay the bills, but zed and gpui are projects people rely on, so I don't think this is acceptable for a core dependency.

looks like something that can be made selectable by cfg flags based on checking this pr

Yes of course it could, but that has certain implications that should be considered. If you add it as a new backend, it will also have to be maintained, so more work for further improvements. This is already a bit of an issue with GPUI given the different renderer implementation. I respect the decision of the zed maintainers to support those implementation, but personally I think we should thrive to consolidate the rendering infrastructure, not further fragment it. There are exciting things for the future that could be enabled (or significantly simplified for end-users) by consolidation, like APIs for custom shaders, direct buffer access for custom elements, embedding into other wgpu apps (or being embeddable by them), like bevy.
