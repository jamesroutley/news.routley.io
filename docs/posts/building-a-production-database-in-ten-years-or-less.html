Caos would parse CaosQL queries, validate them against the current schema, and compile them to equivalent SQL. It was our secret sauce that let us deliver faster and have more fun doing it. Our code relied heavily on metaprogramming. In late 2012, the Python community started looking for volunteers to design a better introspection API (PEP 362). This was something we needed for our RPC library, so we jumped at the chance. This was our first major foray into open-source.

In the post-mortem of this experience, we started looking critically at the stack of tools we‚Äôd been incubating over the past 7 years. Walking back from the office on a brisk Toronto evening, Elvis and I suddenly realized Caos could represent a new way forward. But first it had to stop being an ORM and become a database.

There were several key pieces of intrastructure we needed to build before we could make headway on EdgeDB.

Our plan was to refactor Caos from a Python framework into a full-fledged database server that could handle incoming queries over the network. For this to scale to thousands of simultaneous connections, blocking I/O with threads wasn‚Äôt going to cut it (though that‚Äôs a blog post for a different day). EdgeDB needed an asynchronous core.

Around the same time we designing the API for a hypothetical Python client library for EdgeDB. We quickly got hung up on the transaction API; With yield from, the coroutine syntax at the time, it was painfully verbose:

tx = yield from conn.start_transaction()
try:
    print(yield from tx.query('...'))
except Exception:
    yield from tx.rollback()
    raise
else:
    yield from tx.commit()

We realized this could be elegantly expressed if Python supported async with syntax natively. I started working on a PEP.

async with conn.transaction() as tx:
    print(await tx.query('...'))

Quickly, the proposal expanded to include async for (to support async database cursors) and of course async/await. After a few weeks of fevered work, I published PEP 492, which was finalized in April 2015 and accepted just before the Python 3.5 feature freeze. We started realizing that pure Python wasn‚Äôt fast enough for handling I/O in EdgeDB. The result was uvloop: a drop-in replacement for Python‚Äôs built-in asyncio event loop that was 2-4x faster. We announced it in a rather well-known blog post. In surprisingly short order, uvloop grew to near ubiquity and now powers EdgeDB.

EdgeDB uses Postgres under the hood. Postgres truly is the world‚Äôs most advanced open-source SQL database, and it provides the ultimate foundation on which to build a new and better database abstraction.

This is often a point of confusion among people who are first learning about EdgeDB. If Postgres is already a database, and EdgeDB is built on top of Postgres, then is EdgeDB an ORM (object-relational mapper)?

We don‚Äôt think so. It has its own formally-defined query language, EdgeQL, which matches SQL in power and surpasses it in clarity and brevity. It‚Äôs language-agnostic, whereas most ORMs are libraries written for a particular language. It has its own type system, standard library, binary protocol, client libraries in multiple languages, CLI, workflows, and conventions. By any definition, EdgeDB is a database unto itself.

With async/await and a fast asynchronous event loop, we started evaluating the state of async Postgres drivers for Python. Alas, nothing made the cut, so we built our own. In the process, we learned a lot about Postgres‚Äôs binary protocol and its many vices and virtues.

Armed with async/await, Cython, uvloop, and asyncpg, we had the tools we needed to make EdgeDB a reality. From this point on, we only did consulting to pay the bills. We were laser-focused on releasing the first alpha version of EdgeDB. The technical preview was a great prototype, but we needed EdgeDB to level up before it was ready for a first release.

We formally defined the type hierarchy and tuned the operators and typecasting system for safety and usability. We added built-in support for GraphQL. We designed a binary protocol‚Äîinformed by our asyncpg experience and deep familiarity with Postgres‚Äôs protocol‚Äîand a Python client library that implemented it.
