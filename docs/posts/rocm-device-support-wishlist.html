Please take a look at: #4224

ROCm is currently 29GB installed (vs cuda-toolkit @ 3GB). A large part of this is due to ISA-specific builds, I get it. Additional hardware support is going to balloon this greatly. Barring the creation of a PTX-like IR to solve this, it still seems like there's a bunch of easy wins available w/ not too much work:

  • As mentioned, many officially unsupported GPUs already work via HSA_OVERRIDE_GFX_VERSION or unofficial packages. Why not just provide support for HSA_OVERRIDE_GFX_VERSION in Windows (or automatically apply HSA_OVERRIDE_GFX_VERSION across platforms based on device detection? Add a CI pipeline to test which parts work/don't.
  • You can see people producing 3-6GB docker files w/ limited architecture support; per this suggestion, AMD should contribute slices for chisel to support architecture-specific distributions - honestly, the suggestion of a rocm-install --gfx1102 or something like that seems eminently sensible. AMD should pre-compile and offer all the versions for each piece of hardware, and just allow mix and matching and/or auto-detection of installed hardware.

As I'm sure you know, CUDA provides support for virtually all of Nvidia's GPUs (consumer, workstation, data center) as far back as 2010. I don't think anyone needs that on the AMD side, but having support for everything gfx9*+ seems pretty reasonable to me, assuming you can find a way around forcing everyone to download support for every single GPU (see above).

BTW, I think it's fine to have varying levels of support for each card (just have a link in the compatibility matrix that shows which tests fails for which architectures) and what software works. Most people just want LLM or Stable Diffusion inference. Some want PyTorch and Triton, and some need CK support etc. Make a simple grid for support. Right now, support is represented at a binary 0/1 (which isn't true anyway), but most people are being artificially gated, when their cards actually already work for what they want to use it for.

As a developer, I'd like AMD to offer support for the latest GPUs (/ iGPUs) and widest range of cards. I would also like the Windows and Linux (and WSL) support lists to be the same. I would like support for laptop GPUs because I find working on a laptop more convenient (and everything at my workplace uses them) and I would like iGPU support so that I could buy a laptop with an AMD CPU and an NVIDIA GPU and use it to develop for both CUDA and ROCm (or, more correctly, develop with CUDA and test on ROCm/HIP).

Which leads me to this: I always develop on NVIDIA hardware, because most of the market, on both server and client side, is NVIDIA GPUs. In my current job, where I write something that will run on the cloud or on custom-specced PCs, NVIDIA is naturally an easier choice, because AMD just doesn't offer the ease of development that NVIDIA does. In my previous job, where I wrote consumer software with OpenCL and Direct3D, I found AMD easier and liked it better, but I still spent most of my time on NVIDIA hardware, because that's the market. (Disclaimer: the software I wrote at my previous job didn't run on Linux. It ran Windows, Mac OS, and even Android, to an extent, but it didn't address Linux.)

When it comes to ROCm/HIP, I think that what AMD needs to offer is a path of least resistance, something that would enable developers to address AMD as an afterthought, with as few barriers as possible. In terms of hardware support, this IMO means iGPUs. Ryzen CPUs are quite successful. Desktop ones come with an RDNA 2 iGPU. Laptop ones come with whatever that laptop generation has. A developer using such a platform can have an NVIDIA dGPU and, if AMD supported iGPUs in ROCm/HIP, would still be able to test the AMD part of the software without having to own a separate PC for that purpose.

When it comes to Linux support specifically, I think that the best would be to support the same hardware HIP for Windows offers. If I develop for both Windows and Linux, I'd love to be able to develop on Windows and WSL and have the same GPU run both. With the limited Linux support, this is quite difficult to achieve. Windows supports pretty much all desktop RDNA 2 and 3 cards, which IMO is great, even if I'd love official support for laptop dGPUs (which are supported, even if unofficially) and for iGPUs of these architectures. If AMD doesn't have the resources to support all this on Linux, then at least have architecture-wide support. Do only RDNA 3, but enable it for all RDNA 3 GPUs, so that a 7800 XT, 7700 XT or 7600 would be valid targets development.
