GZGavinZhao

Jan 21, 2025

IMO, a large part of the device support mess can be resolved by providing an alternative ROCm binary release that includes "unofficially" supported GPUs and putting a disclaimer on the lines of "this is not an official AMD product so don't expect support and responsibility from us".

This is because, as mentioned a few comments above, a very large number of devices mentioned are already supported "unofficially", AMD likely just doesn't have the bandwidth to test them thoroughly enough to include them in the release.

By supported "unofficially", I mean that ROCm can run mostly/perfectly fine on those GPUs, using tricks like HSA_OVERRIDE_GFX_VERSION and with patches (e.g. ROCm/Tensile#1897, NixOS/nixpkgs#298388) that the community has been contributing. Some of the most notable "unofficially" supported GPUs are all RDNA1 (gfx101*) and RDNA2 (gfx103*) gpus.

Technical wise, this "unofficial" binary release is definitely doable. Using community patches, Solus provides ROCm binary packages that supports gfx803;gfx900;gfx904;gfx906;gfx908;gfx90a;gfx90c;gfx1010;gfx1011;gfx1012;gfx1030;gfx1031;gfx1032;gfx1033;gfx1034;gfx1035;gfx1036;gfx1100;gfx1101;gfx1102 on a best-effort basis. Someday, I would like to create said "alternative ROCm binary release" that people can get as a drop-in replacement for the official ROCm release, but haven't got enough time for that :(

EDIT: I'm not saying this completely solves the problem. However, given the limited resource that AMD has, providing an unofficial binary release is perhaps the best use of their time. There is only so much engineers can do if the managers/executives want to prioritize something else.

Please take a look at: #4224

ROCm is currently 29GB installed (vs cuda-toolkit @ 3GB). A large part of this is due to ISA-specific builds, I get it. Additional hardware support is going to balloon this greatly. Barring the creation of a PTX-like IR to solve this, it still seems like there's a bunch of easy wins available w/ not too much work:

  • As mentioned, many officially unsupported GPUs already work via HSA_OVERRIDE_GFX_VERSION or unofficial packages. Why not just provide support for HSA_OVERRIDE_GFX_VERSION in Windows (or automatically apply HSA_OVERRIDE_GFX_VERSION across platforms based on device detection? Add a CI pipeline to test which parts work/don't.
  • You can see people producing 3-6GB docker files w/ limited architecture support; per this suggestion, AMD should contribute slices for chisel to support architecture-specific distributions - honestly, the suggestion of a rocm-install --gfx1102 or something like that seems eminently sensible. AMD should pre-compile and offer all the versions for each piece of hardware, and just allow mix and matching and/or auto-detection of installed hardware.

As I'm sure you know, CUDA provides support for virtually all of Nvidia's GPUs (consumer, workstation, data center) as far back as 2010. I don't think anyone needs that on the AMD side, but having support for everything gfx9*+ seems pretty reasonable to me, assuming you can find a way around forcing everyone to download support for every single GPU (see above).

BTW, I think it's fine to have varying levels of support for each card (just have a link in the compatibility matrix that shows which tests fails for which architectures) and what software works. Most people just want LLM or Stable Diffusion inference. Some want PyTorch and Triton, and some need CK support etc. Make a simple grid for support. Right now, support is represented at a binary 0/1 (which isn't true anyway), but most people are being artificially gated, when their cards actually already work for what they want to use it for.
