            Just to have it upfront, I will start with what The Tower Of Weakenings is, and then go back to the motivation and ideas behind it.             Here is a sketch of what the tower looks like under my proposal:

            1. The ‚ÄúClean‚Äù Memory Model: a painfully strict and simple model that you can teach and check.
            2. The ‚ÄúReal‚Äù Memory Model: similar to the ‚ÄúClean‚Äù one, but messier to allow for Useful Crimes.
            3. The Compiler‚Äôs Semantics: whatever random primitives compilers expose, and optimizations they do.

            Yes that sure didn‚Äôt explain anything at all. That was actually ‚ÄúWhat Should I Do?‚Äù, which is actually the important thing. Did you follow all 6 steps? Ok great, here‚Äôs an actual explanation:

            Under Strict Provenance (which is all playing pretend), if you cast an integer to a pointer, that‚Äôs ok, it‚Äôs fun to play pretend, but that pointer is invalid. That means reading/writing through that pointer or using offset on it is Undefined Behaviour. (wrapping_offset is fine, and the usual ZSTs Get To Lie rules apply.)

            If you‚Äôre doing FFI, just make sure that the API is ‚Äúright‚Äù from Rust‚Äôs perspective. This may involve tweaking extern decls that lie about pointers.

            If you‚Äôre doing MMIO stuff, you‚Äôre basically outside the memory model already but we‚Äôre working on having a nicer story for you too, but this is too far outside my domain of expertise to help with.

            The golden rule is that the union of int | ptr is ptr, not int. ABIs generally all agree that pointers and pointer-sized-integers should have identical ABIs, because this punning is super common‚Ä¶ unless your on an m68k, which thinks you‚Äôre a sinner either way (for return values). God, nothing can be simple, can it?

            Please also do this for AtomicUsize and AtomicPtr.

            That‚Äôs it. Just say things that are pointers are pointers.

            Without proper memory models formalizing things like ‚Äúwhat is a pointer‚Äù, compiler backends can very innocently introduce several optimizations that are all fine in isolation but can rube-goldberg machine into miscompilations when combined.

            These situations are especially nasty because you can‚Äôt obviously just point at one optimization and say ‚Äúoh it‚Äôs that one, that one is bad‚Äù. How do you decide which seemingly-innocent optimization is a problem? This is what a memory model is for.

            You define an ‚ÄúAbstract Machine‚Äù that your language is ‚Äúemulating‚Äù the semantics of, and then optimizations which change the observable behaviour of ‚Äúthe Abstract Machine‚Äù are ‚Äúbad‚Äù. This is basically the same situation as a SNES emulator glitching out and flipping Mario upside-down. You generally want your emulator to faithfully reproduce the game!

            Conversely, having a memory model also defines the scope of what programmers are allowed to expect to work, and the compiler doesn‚Äôt have to care about somehow trying to faithfully reproduce things that don‚Äôt work on the Abstract Machine. Returning to SNES emulators, this is like refusing to support romhacks that do things that wouldn‚Äôt work on the actual hardware, because you have no idea what on earth that even means. Harsh, but fair!

            Things that ‚Äúcan‚Äôt work‚Äù on the Abstract Machine are Undefined Behaviour. One thing we generally all agree should definitely be Undefined Behaviour is Use After Frees, because it makes no sense, is non-deterministic, and is a huge source of security vulnerabilities.

            Ok so let‚Äôs make that rule part of our Abstract Machine.

            ‚Ä¶how do you do that? Can you do that??

            Let‚Äôs think it through:

            1. Allocate some memory
            2. Get a pointer into that memory
            3. Free the memory (dangling pointer)
            4. Allocate some more memory
            5. GET LUCKY and have your dangling pointer point into it
            6. Read through the pointer (use after free)

            How do you define a machine where this isn‚Äôt ‚Äúallowed‚Äù? One option is to define the machine to never reuse addresses, so you can never Get Lucky. This is a real thing some systems try to do, but most systems just aren‚Äôt like that. Is that a problem? Are we breaking our promise to faithfully emulate the Abstract Machine if we reuse allocations? You can‚Äôt reliably observe it, but whenever you Get Lucky you can. But maybe that‚Äôs fine because it‚Äôs Undefined Behaviour to dereference the pointer, so it‚Äôs the programmer‚Äôs fault if they observed it! Yeah!

            Wait, does that mean it has to be Undefined Behaviour to check if a ‚Äúdangling‚Äù pointer is equal to another one? Oof, saying pointer equality is UB is rough. Like sure we can all agree inequalities between pointers from different allocations should be unspecified, that‚Äôs Totally Natural And Fine, but equality? Say it ain‚Äôt so!

            ‚Ä¶Ok so you can see how this gets really wonky really fast right?

            As I understand it, the consensus solution to this is to just admit that stuff like banning Use After Frees implies pointers are kinda magic, and you just say they ‚Äúknow‚Äù the allocation they point into and aren‚Äôt allowed to access any other memory. Use After Frees are then forbidden on the premise that a reallocation is a ‚Äúdifferent‚Äù allocation, and so any dangling pointers that happen to point into it can‚Äôt access it. No Getting Lucky. Well, you can get lucky enough to notice two pointers are ‚Äúrandomly‚Äù equal, but you don‚Äôt get to do anything scary with that information because your dangling pointer is still ‚Äúinvalid‚Äù.

            The idea that pointers just ‚Äúknow‚Äù what allocation they‚Äôre part of is provenance. So, cool, pointers have provenance in the Abstract Machine, problem solved!

            ‚Ä¶huh? What‚Äôs that? You want to turn a pointer into an integer? Ok well that‚Äôs f- AND TURN IT BACK INTO A POINTER? Oh. My. Um. Did you want all integers to have provenance too? That has some very large implications for like‚Ä¶ x==y not letting you substitute x for y anywhere in your optimizer, because they might have different provenance even if they compare equal. Yeah that sounds horrible to me too.

            Ah ok, but the C folks seem to have cracked this one. If you hop on over to this 123 page paper, you‚Äôll see the PNVI-ae-udi model, which at a high level roughly says:

            • The Abstract Machine has a global list of all ‚Äúexposed‚Äù allocations
            • Allocations come into the world unexposed (unaliased)
            • Casting a pointer to an integer exposes the allocation for that pointer‚Äôs provenance
            • Casting an integer to a pointer does a global hittest on all exposed allocations
            • If you hit an exposed allocation, congrats! You get provenance to it.
            • If you hit two exposed allocations (because of one-past-the-end), be sad and tell the programmer to ‚Äújust be consistent‚Äù and only access one of them.

            This makes a lot of sense! Basically it‚Äôs saying if you cast a pointer to an integer, the compiler has to throw up its hands in disgust and just accept that any integer gets to be cast into a pointer into that allocation. That‚Äôs‚Ä¶ pretty tame? Assuming you don‚Äôt need to allow other operations to expose.

            But also‚Ä¶ it lets you write this:

            1. Allocate some memory
            2. Get a pointer into that memory
            3. Cast it to an integer (A)
            4. Free the memory
            5. Allocate some more memory
            6. Also expose that one somehow
            7. Cast (A) back into a pointer
            8. GET LUCKY and have your pointer now get blessed with provenance over the new allocation

            This is just a Use After Free with extra steps! Trying to allow for int-to-ptr casts just reintroduces the concept of Getting Lucky, and that sucks. It might be a necessary evil, but it still makes me sad.

            It also makes sanitizers really sad. The more you crank up ‚Äúthings that are allowed to expose addresses‚Äù (Hmm, doesn‚Äôt memcpy operate on char*? Isn‚Äôt that an integer..?), the more flexible a sanitizer needs to be about things that Should Definitely Be Undefined Behaviour, But I Have To Assume You‚Äôre Very Smart And Perfect Because Integer Casts Say So. Which is to say, you can have a dynamic execution that everyone 100% agrees should be a Use After Free, and which you would therefore hope a sanitizer to catch‚Ä¶ and it just has to shrug it‚Äôs shoulders and assume it isn‚Äôt.

            On the flip side, if you say ‚Äúscrew that‚Äù, your sanitizer will constantly set off false alarms because everyone is told that int-to-ptr casts are totally fine and cool. This also sucks!

            This whole mess is why Strict Provenance is such a nice simplification to a memory model: it just puts it foot down and says No Absolutely Not You Do Not Get To Get Lucky.

            But of course, Strict Provenance is ‚Äúa lie‚Äù, so you still can get lucky, so what‚Äôs the point? The point is that if everyone agrees to just pretend that you‚Äôre not allowed to do int-to-ptr casts, then having our sanitizers default to the strict semantics is actually useful and effective! Programmers that opt into this game also don‚Äôt have to worry about ‚Äúare we doing PNVI-ae-udi‚Äù or something else, because‚Ä¶ it doesn‚Äôt matter! Tower Of Weakenings Baby! Allowing any kind of int-to-ptr shenanigans is definitely weaker than forbidding them completely!

            But ok we have all this code that does pointer crimes, this is useless! It would be, if we didn‚Äôt have the glorious hack innovation of with_addr! This is the part of strict provenance that makes everything ‚Äúwork‚Äù:

            fn with_addr(self, addr: usize) -> Self

            This is basically the exact same thing as usize as ptr BUT instead of being a unary operation, it‚Äôs binary. You need to bring along a pointer with the provenance you think your new pointer should have, and then we use it to properly ‚Äúrebuild‚Äù the pointer, unambiguously re-establishing the provenance chain of custody, hooray! Toss in some conveniences like map_addr and you‚Äôve got yourself an actually tolerable system for still doing tagged pointer crimes while staying in the confines of Strict Provenance:

            unsafe {
    
    static HAS_DATA: usize = 0x1;
    static FLAG_MASK: usize = !HAS_DATA;

    
    let my_precious_data: u32 = 17;
    assert!(core::mem::align_of::<u32>() > 1);

    
    let ptr = &my_precious_data as *const u32;
    let tagged = ptr.map_addr(|a| a | HAS_DATA);

    
    if tagged.addr() & HAS_DATA != 0 {
        
        let data = *tagged.map_addr(|a| a & FLAG_MASK);
        assert_eq!(data, 17);
    } else {
        unreachable!()
    }
}

            Since the MVP shipped, we‚Äôve made a couple changes. Those are my ‚Äúrelease notes‚Äù for the change, but here‚Äôs a summary:

            Ralf decided it would be best to make it more explicit as to whether you‚Äôre actually playing this game or not. In his PR he added APIs that explicitly use the ‚Äúexpose‚Äù terminology and some vague gesturing to the same kind of thing the C proposal is doing ‚Äì which is basically all we would have been able to tell you before strict provenance too.             • ptr::from_exposed_addr(usize) -> *const T
            • ptr::from_exposed_addr_mut(usize) -> *mut T

            And these are 100% semantically identical to ptr as usize and usize as ptr, while also actually telling us that you understand this is a thing and you are relying on.             I asked him to explicitly mark in the docs that this UB is not actually exploited right now and if this ends up being a mess we‚Äôre willing to weaken it back to be identical to expose_addr. The motivation for marking it UB is twofold:

            1.             2. It justifies the kind of optimizations llvm already does today, which are definitely unsound absent any proper provenance model. Making addr explicitly not expose the allocation makes that transform sound!
