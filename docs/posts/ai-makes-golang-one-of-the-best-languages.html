AI Makes Golang One of the Best Languages

Apr 20, 2026

If there is one thing I've learned working with agentic coding tools like Claude Code, Codex, and Cursor it's this:

Large language models amplify inconsistent technology and ecosystems while reinforcing consistent ones.

Languages with a fragmented ecosystem fair much worse with LLMs than languages with a cohesive one. Golang is one ecosystem which (thanks to Google) benefits from its mundane stability.

The most popular programming language in the world - the one for the web - has always been a hot mess since its inception. There are hundreds of frontend frameworks with major players contending for adoption: Next.js, React, Svelte, Angular, etc.

The State of JS survey found only three libraries (React, Jest, and Vite) captured over 50% of respondent usage while maintaining >50% retention — the survey itself calls this "a sign of a relatively fragmented ecosystem."

Python, an equally successfuly language, brings a long history of instability. The Python 2 to Python 3 transition took nearly 20 years end-to-end. Not the mention dozens of tools which accomplish varying degrees of consistency:

Are you using default pip3, poetry, uv? Which toolchain and whether you're cross compiling matters — fourteen tools is at least twelve too many, and xkcd 1987 was a joke in 2018 that's still funny. Are you using Python async or a task queue instead? Django or FastAPI?

Astral was acquired by OpenAI largely for their talent but also the rock-solid engineering and unification they've done to normalize the insanity of Python dependency resolution (beware, here be dragons).

For those of us who build and maintian complex systems managed by dozens or hundreds of engineers know, the flexibilties and fragmentations of a language become liabilities in productions. Merits aside, it's nice when there's just one correct, fast, and predictable way to do things...

Highly consisteny data in a training corpus yields better results during inference. It should be no surprise that if I train on some text and certain patterns are almost always followed (e.g. noun then verb), then we should expect the model to encode this relationship strongly.

Programming languages are no different. High cosine similarity in the training data strengthens the model's ability to output likely tokens in the attention and MLP layers of the transformer. Typed, compiled, language with first-party standard libraries benefit the most. See a pattern, repeat a pattern.

In the dynamic world, LLMs understnad Rails much better than JavaScript frameworks. There are at least a dozen production grade JavaScript frameworks offering differing combinations of Rails' core features. Convention over configuration was a win for human programmers as much as machines — as DHH put it at OSCON 2004, "flexibility is overrated… constraints are liberating."

Golang best represents convention, annoyingly so...

err := work()
if err != nil {
	return nil // i hate writing this, LLMs don't care
}

Golang balantly pissed over 20+ years of programming language theory to build a verbose, annoying, unexpressive language - and it worked.

Frustraitingly, Rob Pike (one of the creators of Golang) is quoted from his Lang Next 2014 talk entitled From Parallel to Concurrent:

The key point here is our programmers are Googlers, they’re not researchers. They’re typically, fairly young, fresh out of school, probably learned Java, maybe learned C or C++, probably learned Python. They’re not capable of understanding a brilliant language but we want to use them to build good software. So, the language that we give them has to be easy for them to understand and easy to adopt.

While pragmatic, there a sense of disdain for certain types of programmers. "Here, you're be writing with crayons today".

I've personally written hundreds of thousands of lines of Golang over my career. As a programmer writing code by hand I found Golang to be infuriating. As an engineer enabled by AI, Golang is heaven and I have fallen back in love with it.

LLMs make the drudgery go away. Out of the box Golang gives you (and your agents) a massive leg-up on coding:

Concurrency out of the box - goroutines are much more powerful a primitive for coding agents than any other concurrency. They're simple, type-safe, and ubiquitously used. No what color is your function problems — Nystrom called it a decade ago: "Go is the language that does this most beautifully… Go has eliminated the distinction between synchronous and asynchronous code."

// Fan out N HTTP fetches, collect results. No async, no Promise.all.
results := make(chan string, len(urls))
for _, u := range urls {
    go func(u string) {
        resp, err := http.Get(u)
        if err != nil {
            results <- err.Error()
            return
        }
        defer resp.Body.Close()
        results <- resp.Status
    }(u)
}
for range urls {
    fmt.Println(<-results)
}

A std library backed by Google. The default net/http package alone probably runs double-digit percentages of the internet's micro-services. Google also funds and backs the crypto package — world-class cryptography libraries which I've used in production systems at Zoom and Keybase.

// A production HTTP server. Zero dependencies outside the stdlib.
package main

import (
    "fmt"
    "net/http"
)

func main() {
    http.HandleFunc("/healthz", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintln(w, "ok")
    })
    http.ListenAndServe(":8080", nil)
}

Strong compiler & toolchain with only one right way to do things: go fmt, go vet, and now go fix. The best possible combination for an LLM is a consistent corpus of non-breaking code to train on plus 'one right way' tooling for testing and development. gopls is a fantastic guardrail for agents for realtime feedback; additionally golangci-lint allows you to select coding styles and primitives you want to enforce statically — no prompting necessary.

$ go vet ./...
./user.go:22:2: result of fmt.Errorf call not used
./user.go:38:9: declaration of "err" shadows declaration at line 34

$ golangci-lint run
user.go:51:6: exported func LoadUser should have comment (revive)
user.go:63:3: if block ends with return, drop this else (golint)

Every line is a deterministic, file:line:column signal — exactly the shape an agent can loop on until the diagnostics are empty.

Native-like performance with garbage collection. LLMs prove to be inconsistent in managing memory. Tools like Rust enforce memory safety at the type and borrow checking level. Writing C/C++ with a coding agent is hard because the training data contains lots of memory bugs, use after free, etc.

// Allocate, return, forget. No malloc/free, no Box, no lifetimes.
func parseLines(r io.Reader) []string {
    var out []string
    s := bufio.NewScanner(r)
    for s.Scan() {
        out = append(out, s.Text())
    }
    return out
}

Small but known set of footguns in Golang. Nil pointers are difficult to trace down for human engineers in production stack traces; agents are quite good at this given the right tools. Errors are values, and the shape never varies — an LLM has seen this pattern ten million times.

// One shape. No try/catch, no Result<T, E>, no Maybe.
data, err := os.ReadFile(path)
if err != nil {
    return fmt.Errorf("read config %q: %w", path, err)
}

Reading this, you should think to yourself: oh, this language and toolchain can write the majority of non-visual software I would want. Things from scripts that process data in parallel, CLIs, web servers, and TUIs. For all the same reasons Java is/was ubiquitous, Golang fixes what Java introduced while expanding on the myriad of locations it can run. GitHub's 2025 Octoverse report puts it plainly: the rise of typed languages suggests AI isn't just changing the speed of coding, but also influencing which languages teams trust to take AI-generated code into production.

The next CLI, backend server, or agent orchestrator can and likely should be written in Golang using a coding agent. The language is simple with strong primitives; it progressively discloses primitives like concurrency while removing the necessity to debug memory handling directly.
