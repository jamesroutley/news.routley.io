Why Differential?

Monolithic codebases don't have to result in monolithic services.

Split your services without splitting your codebase.

Differential lets you keep all your code co-located, and maintain multiple entry-points that can bootstrap parts of the codebase independently at run time. This means more reusable code, and less tooling and complexity for a service-oriented architecture.

Type-safe service interfaces, without the overhead.

Differential SDK utilizes Typescript types from plain old functions to automatically infer the service definition for remote procedure calls. This means no compile steps, sharing Open API specs, or generating Protobufs as an intermediate representation.

Network Comms and Serialisation - taken care of.

All your functions communicate through the Differential control-plane. This makes it possible for it to abstract away network communication and serialization complexities. Open-source codebase allows you to look under the hood, or self-host on your own infrastructure.

FAQ

Is Differential a RPC framework?

Close. It's an RPC framework and a service mesh that works together. The services don't communicate directly with each other. Instead, it uses the control-plane as a broker. The control-plane implements a job queue that load balances and orchestrates all the function calls.

How do I define the service contract? Do you use something like Protobufs / Open API Spec?

In short, you don't. When you build services with Differential, service contracts are inferred by function signatures. Therefore, you don't need to write your typed code and build a separate service interface as you'd do in a REST / gRPC / tRPC service. This way the entire service interface is expressed in code.

Can you ELI5 the difference between Differential and REST?

With traditional REST services, you define controllers (functions) and expose them through routes, mapping their outcomes to HTTP Status Codes and error messages. This involves setting up a server for TCP traffic, with authentication to restrict access to your clients. You then build and deploy the client separately.

How secure is the control-plane, given all my data is going through it?

Fair concern. Differential operates on your compute. Control-plane is an orchestrator. Client SDKs have the capability to encrypt the data using a pre-determined encryption key. This key is never sent to the control-plane. Clients can encrypt and decrypt payloads seamlessly, as long as they have access to the encryption keys via configuration.
