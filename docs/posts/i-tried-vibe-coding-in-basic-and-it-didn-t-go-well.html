I tried vibe coding in BASIC and it didn't go well

With the rise of LLM systems (or “AI” as they are annoyingly called), the term “vibe coding” is all the rage recently. Vibe coding is when you rely almost entirely on these “AI” system to write your code for you via a series of drawn-out conversations.

Putting on my cranky old man hat, I find most of it rather frustrating. Now I’m not some retro-sycophant who refuses to use modern tools. I’ve been a software professional for over 30 years at this point and I’ve used lots of tools throughout the years. Many that were cool and hip have long since disappeared or fallen out of fashion. I’ve also actually used these AI systems myself to help figure out some code. I find they function really well as a better search engine for documentation and can also serve as a way to “rubber duck” ideas and get some outlines. Any code I get back rarely seems to work as-is, but it often can provide ideas for something I hadn’t considered or perhaps point out something I did not know about.

After all, today’s technology is complex and it is not always easy to find what you need in the documentation.

But how does vibe coding work with something far simpler, such as the simplest programming language of them all: BASIC?

I suspect this might be a little tricker than it first appears. First of all, usage of BASIC largely predates the internet so there is far less BASIC code out there for LLMs to learn from. Many online retro BASIC books and docs are often scanned copies rather than easily digested plain text. Also, BASIC is not a single language, but more of an umbrella term for a family of languages all of which have the same “vibe”, but often very different commands and capabilities, some of which rely heavily on PEEKs and POKEs to really take advantage of their system’s features. And those ancient forms of API calls are poorly documented online.

So this should be an interesting challenge and to make it even slightly trickier, I’m going to use Atari BASIC rather than a much more common Microsoft BASIC variant, such as AppleSoft. Why? Because this is an Atari newsletter and I have plenty of Atari equipment to test things with. Alas, I don’t have any Apple II-era systems.

With that said, I will be using the Atari800MacX emulator because it makes it easy to paste in the generated code.

Now with all that out of the way, which LLM AI system should we try? Considering ChatGPT is the most commonly known and one most are familiar with, that’s I’m using for this experiment, although Anthropic Claude is also great for use with coding.

To do proper vibe coding, I need to refrain from correcting the generated code or even telling the AI what exactly is wrong.

So what should be do first? Let’s start with a typical first BASIC program, the precursor to “Hello World”.

Prompt: Please write me a simple program in Atari BASIC that repeatedly prints “Hello, World” to the screen.

ChatGPT aced this simple test. I have to say, I do love seeing “GOTO 10” like that!

Atari BASIC does not have a built-in way to draw a circle so let’s ask for that:

Prompt: Write me an Atari BASIC program that draws a blue circle in graphics mode 7.

So far, the vibes are flowing. This program ran perfectly, although it only drew a rough, dotted circle.

I asked it for a solid outline instead of a dotted outline and it came back with this code:

Continuing the graphics theme, let’s take it up a notch. Atari BASIC does have built-in commands for drawing lines and accessing the joystick, which are more than enough to make a simple drawing program.

Prompt: Write an Atari BASIC program that lets you draw on the screen using a joystick. It should use graphics mode 7 and allow you to rotate between colors by pressing the fire button.

The resulting code had several errors:

To use COLOR as a variable name, LET is required. The “;” is not used for comments in any version of BASIC I’ve ever seen.

Prompt: Line 30 shows this error: ERROR- COLOR=1. There are also errors on any lines that use ";" for a comment.

It fixed these errors by changing COLOR to just C and getting rid of most comments:

That code ran fine, but the joystick directions were wrong. Joystick up draws right; down draws left; left draws up; right draws down.

How about a simple game? Using sprites (player/missiles) in Atari BASIC requires a machine language subroutine, so that’s probably not realistic. Other types of games use a redefined character set which only require some PEEKs and POKEs. We can ask ChatGTP to make a rudimentary version of Space Invaders this way and that seems worth a try. With the game being so well known, hopefully I don’t need to provide a ton of prompting.

Prompt: Create a simple version of Space Invaders using Atari BASIC. Use a large text graphics mode with a redefined character set for the aliens, spaceship and missiles. The player’s ship should be controlled with the joystick.

It’s first response was rather simple:

That code did run without any errors, but it didn’t really work. There was a ton of flicker and the aliens scrolled off the top of the screen. Moving the ship and firing did work, though.

I then suggested that it try to use Graphics 1 and that it not redefine any characters and instead use “A” for alien, “S” for ship and a vertical bar for the missile.

The code that came back ran OK, but it was still using Graphics 0.

That is because it is using PRINT instead of PRINT #6. I told ChatGPT that it was still using Graphics 0, but it was not able to fix it on its first revision.

I again told it that it was still running in Graphics 0 to see if it might figure it out. But it did not figure it out, instead going off on a tangent about screen memory and character sets, neither of which were relevant.

I decide to tell it what was wrong: The problem is that you are using PRINT instead of PRINT #6;. You don't need to alter memory or character sets. Please try again.

It came back with better code, but it took several back-and-forths before it came back with code that actually ran. And even then it later crashed with an ERROR 141, Cursor out of range because it was using INPUT #6, which doesn’t make sense here.

Instead it really should use LOCATE to check for a collision with an alien ship.

Frankly, this vibe coding was exhausting and I gave up trying to coerce ChatGPT into getting this working. I think ChatGPT gave me a reasonable program outline, but the back-and-forth was just tedious! I certainly could have fine-tuned things faster.

So I did. Here is the working code for this simple game:

10 GRAPHICS 1+16
20 POKE 77,0:REM Turn off text cursor
30 SHIPX=10:FIRE=0:MX=0:MY=0
40 REM === DRAW ALIEN ROW ===
50 FOR I=2 TO 18 STEP 3
60 POSITION I,5:PRINT #6;"a";:NEXT I
100 REM === MAIN LOOP ===
110 ST=STICK(0):BT=STRIG(0)
120 IF ST=11 AND SHIPX>0 THEN SHIPX=SHIPX-1
130 IF ST=7 AND SHIPX<19 THEN SHIPX=SHIPX+1
140 POSITION 0,22:PRINT #6;"                    ";
150 POSITION SHIPX,22:PRINT #6;"S";
160 IF FIRE=0 AND BT=0 THEN FIRE=1:MX=SHIPX:MY=18
170 IF FIRE=1 THEN GOSUB 300
180 GOTO 100
300 REM === MOVE MISSILE ===
310 LOCATE MX,MY,C
315 POSITION MX,MY:PRINT #6;"."
320 FOR D=1 TO 50:NEXT D
330 POSITION MX,MY:PRINT #6;" ";
340 MY=MY-1
350 IF MY<0 THEN FIRE=0:RETURN 
355 IF C<>32 THEN S=S+10:FIRE=0
370 POSITION 0,0:PRINT #6;S
390 RETURN

There’s not all that much here. You can use the joystick to move the “S” ship left/right at the bottom. Press fire to shoot a “.” as the missile. If you hit an “A” alien then you get 10 points. The aliens don’t move. As I said, it’s rather simple.

As simple as this is, I’m really not sure how someone that doesn’t know Atari BASIC would be able to get it working at all doing just vibe coding with ChatGPT, which would constantly go back to writing incorrect code. Even when I knew something was wrong and corrected it, it didn’t always seem to help much. It feels like you’d hit an endless loop of back-and-forth prompts that won’t ever work properly.

As I mentioned at the beginning perhaps this is more useful with modern languages, but I’d say with an old-school BASIC, vibe coding was ultimately a bust.
