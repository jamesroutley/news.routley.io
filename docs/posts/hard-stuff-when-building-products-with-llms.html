Hard stuff when building products with LLMs

Earlier this month, we released the first version of our new natural language querying interface, Query Assistant. People are using it in all kinds of interesting ways! We‚Äôll have a post that really dives into that soon. However, I want to talk about something else first.¬†

There‚Äôs a lot of hype around AI, and in particular, Large Language Models (LLMs). To be blunt, a lot of that hype is just some demo bullshit that would fall over the instant anyone tried to use it for a real task that their job depends on. The reality is far less glamorous: it‚Äôs hard to build a real product backed by an LLM.

Here‚Äôs my elaboration of all the challenges we faced while building Query Assistant. Not all of them will apply to your use case, but if you want to build product features with LLMs, hopefully this gives you a glimpse into what you‚Äôll inevitably experience.

A quick overview of Query Assistant

At a high level, Query Assistant allows you to express a desired Honeycomb query in natural language, such as:

  • Which service has the highest latency?
  • What are my errors, broken down by endpoint?
  • Which users have the highest shopping cart totals?

In return, you get a Honeycomb query that executes as a best effort ‚Äúanswer‚Äù to your natural language query. The idea isn‚Äôt that it‚Äôs perfect, but it‚Äôs better than nothing‚Äîand it‚Äôs easy for you to refine what comes back using our Query Builder UI.

How it works under the hood

Query Assistant is all about prompting, which assembles a task and data/context as input to an LLM. In no particular order, we pass the following things:

  • The user‚Äôs input where they ask for a query in natural language
  • Information about what constitutes a Honeycomb query (visualization operators, filter operators, the structure of different clauses in a query, etc.)
  • Information about the domain of instrumentation data (e.g., trace.parent_id does-not-exist refers to a root span in a trace, and is often used to represent a request)
  • The schema that a query needs to be produced for (since you need real columns to choose to plug into a query)
  • Several examples, similar to a few-shot prompt
  • The existing query (if it exists)
  • Instructions on what to do

And that‚Äôs it! Behind the scenes, we take output from an LLM, parse it and correct it (if it‚Äôs correctable), and then execute the query against our query engine. We don‚Äôt plug this into a chat UI‚Äîwe think that's the wrong interface for us. In fact, we think no interface is the right interface. Aside from the textbox and button to accept natural language input, everything else is just the same Honeycomb UI.

Context windows are a challenge with no complete solution

I casually mentioned that we use ‚Äúthe schema that a query needs to be produced for‚Äù in our prompt for an LLM. Unfortunately, there‚Äôs nothing casual about it. LLMs have a limit to the amount of input that can accept. That limit, called a context window, includes everything: your inputs, all possible outputs of the LLM, and any data you want to pass to it.

Because we made Query Assistant available to everyone, we needed to have an approach for dealing with context that‚Äôs bigger than the context window. Some customers have schemas with over 5000 unique fields, and there‚Äôs no way for us to know up front which subset is the ‚Äúcorrect‚Äù one to select. So we considered several approaches:

  • Turn off the feature for customers with ‚Äúbig schemas,‚Äù or at least turn it off for only those schemas
  • Chunk up a big schema and make N concurrent calls to an LLM with some notion of a ‚Äúrelevancy score,‚Äù pick the best one, and hope that the boundaries between chunks don‚Äôt elide critical information
  • Chain LLM calls by repeatedly building and refining a query with subsets of a schema, with the hope that after N serial calls you end up with something relevant
  • Use Embeddings and pray to the dot product gods that whatever distance function you use to pluck a ‚Äúrelevant subset‚Äù out of the embedding is actually relevant
  • Find other ways to get creative about pulling in a subset of a schema

We decided to find other ways to get creative, although we‚Äôll likely use Embeddings in the near future.¬†

As it turns out, people generally don‚Äôt use Honeycomb to query data in the past. In fact, when you constrain a schema to only include fields that received data in the past seven days, you can trim the size of a schema and usually fit the whole thing in gpt-3.5-turbo‚Äôs context window.

However, even constraining a schema by time isn‚Äôt enough for some customers. In some cases we still need to truncate the number of fields we use, resulting in a hit-or-miss experience depending on if the most relevant fields in a schema were truncated or not. We‚Äôre looking into the right prayers for the dot product gods with Embeddings to help with this, since it seems to be the most tractable alternative approach. Spoiler alert: the dot product gods aren‚Äôt always right, so we‚Äôre probably going to have to test in prod for this one and see if it‚Äôs an overall improvement when activated more broadly.

There are promising advancements in models with very large context windows. However, in our experiments with Claude 100k, it‚Äôs several times slower if we dump a full schema into our prompt, and it hallucinates more often than if we use an Embedding to pluck out a smaller, more relevant subset of fields. Maybe that will get fixed in time, but for now, there‚Äôs no complete solution to the context window problem.

LLMs are slow and chaining is a nonstarter

Commercial LLMs like gpt-3.5-turbo and Claude are the best models to use for us right now. Nothing in the open source world comes close. However, this only means they‚Äôre the best of available options. They can take many seconds to produce a valid Honeycomb query, with latency ranging from two to 15+ seconds depending on the model, natural language input, size of the schema, makeup of the schema, and instructions in the prompt. As of this writing, although we have access to gpt-4‚Äôs API, it‚Äôs far too slow to work for our use case.

If you google around enough, you‚Äôll find people talking about using LangChain to chain together LLM calls and get better outputs. However, chaining calls to an LLM just makes the latency problem with LLMs worse, which is a nonstarter for us. But even if it wasn‚Äôt, we have the potential to get bitten by compound probabilities.

Let‚Äôs imagine an LLM and prompt that produces a valid Honeycomb query for 90% of all inputs. That‚Äôs quite good! However, if you need to chain calls to that LLM together, then that can potentially result in less accuracy, because‚Ä¶ math. A 90% accurate process repeated 5 times is (0.9*0.9*0.9*0.9*0.9), or 0.59, 59% accurate. Ouch. Fortunately, there are ways to mitigate and improve this process by tweaking the prompts that you chain together, and in practice, it doesn‚Äôt result in such a steep drop-off in accuracy.

We found no tangible improvements in the ability to generate a Honeycomb query when chaining LLM calls together. The book isn‚Äôt closed on this concept altogether, but here‚Äôs your warning: LangChain won‚Äôt solve all your life‚Äôs problems.

Prompt engineering is weird and has few best practices

As mentioned earlier, the way Query Assistant works today is through prompt engineering. Prompt engineering the art and science of getting a ML model to do useful stuff for you without training it on particular data and/or expected outputs. And here‚Äôs the thing: it‚Äôs the wild fuckin‚Äô west out there. Just look at all the techniques in the link to see what wild and interesting stuff people try with prompting. Here‚Äôs some things we tried:

  • Zero-shot prompting: didn‚Äôt work
  • Single-shot prompting: worked, but poorly
  • Few-shot prompting with examples: seems to work well
  • ‚ÄúLet‚Äôs think step by step‚Äù hack: less likely to produce a query for more ambiguous inputs
  • Chain of thought prompting: unclear; not enough time to validate

There are improvements we can make to our prompts by combining some of the emerging prompting techniques available. However, we wanted to deliver something fast, and experimenting with prompting is a time consuming process. It‚Äôs hard to evaluate the effectiveness of a prompt for us because we have an interesting constraint to be correct and useful for broad inputs.

Correctness and usefulness can be at odds

Earlier, I said that we use an LLM to produce a Honeycomb query. That query needs to be correct to be used, but that‚Äôs not the whole story. We must be able to do two things beyond simply producing a correct query:

  1. Accept broad, potentially ambiguous inputs from users
  2. Produce a query that‚Äôs ‚Äúhelpful‚Äù based on certain behaviors we know about Honeycomb

As we‚Äôve learned from shipping our product, our users input every possible thing you can imagine. We get queries that are extremely specific, where people more-or-less type out a full Honeycomb query in English, even using the terminology in our UI. We also get queries that literally say ‚Äúslow‚Äù and nothing else.¬†

Clearly, no prompt + LLM combination can produce a Honeycomb query for all possible inputs, especially if those inputs are extremely vague (how on earth should we interpret slow?!). However, it‚Äôs unhelpful for us to be pedantic. What we think is vague may not be vague to someone using the tool, and our hypothesis is that it‚Äôs better to show something than nothing at all. And so our prompt needs to work with inputs that might not make much sense.

Supporting very broad inputs is the area where a supposed improvement to prompting techniques, zero-shot chain of thought prompting, seemed to make the LLM behavior ‚Äúworse.‚Äù In testing, a zero-shot chain of thought prompt reliably failed to generate a query at all when inputs were vague. And based on data we have about what people ask Query Assistant, going live with this would have been a mistake, since we get a lot of vague inputs.

Additionally, just doing what someone asks for isn‚Äôt always the right thing.

For example, we know that when you use an aggregation such as AVG() or P90(), the result hides a full distribution of values. We‚Äôve found countless times with customers that while aggregations are fine to show a general trend, the fact that they hide a full distribution of values means that you can easily miss problems in your systems that become bigger problems later on. In this case, you typically want to pair an aggregation with a HEATMAP() visualization.

Unfortunately, accepting broad inputs and needing to apply some form of ‚Äúbest practice‚Äù on outputs really throws a wrench into prompt engineering efforts. We find that if we experiment with one approach, it improves outputs at the cost of accepting less broad inputs, or vice-versa. There‚Äôs a lot more work we can do to improve our prompting, but there‚Äôs no apparent playbook we can just use right now.

Prompt injection is an unsolved problem

If you‚Äôre unfamiliar with prompt injection, read this incredible (and horrifying?) blog post that explains it. It‚Äôs kinda like SQL injection, except worse and with no solution today. When you connect an LLM to your database or other components in your product, you expose all of these parts of your product (and infrastructure) to prompt injection. We took the following steps that we think can help:

  • The output of our LLM call is non-destructive and undoable
  • No human gets paged based on the output of our LLM call
  • The LLM isn‚Äôt connected to our databases or any other service
  • We parse the output of an LLM into a particular format and run validation against it
  • By not having a chat UI, we make it annoying and difficult to ‚Äúexperiment‚Äù with prompt injection inputs and seeing what outputs get returned
  • Our input textbox and allowed outputs are truncated
  • We have rate limits per user, per day

If someone is motivated enough, none of this will stop them from getting our system to do something funky. That‚Äôs why we think the most important thing is that everything we do with an LLM today is non-destructive and undoable‚Äîand doesn‚Äôt touch user data. It‚Äôs also why we‚Äôre not currently exploring a full chat UI that people can interact with, and we have absolutely no desire to have an LLM-powered agent sit in our infrastructure doing tasks. We‚Äôd rather not have an end-user reprogrammable system that creates a rogue agent running in our infrastructure, thank you.¬†

And for what it‚Äôs worth, yes, people are already attempting prompt injection in our system today. Almost all of it is silly/harmless, but we‚Äôve seen several people attempt to extract information from other customers out of our system. Thank goodness our LLM calls aren't connected to that kind of stuff.

LLMs aren‚Äôt products

There‚Äôs a lot of ‚Äúproducts‚Äù out there that are just a thin wrapper around OpenAI‚Äôs completions API with a barebones degree of ‚Äúcontext‚Äù or ‚Äúmemory‚Äù (usually via Embeddings). These will all likely disappear by the end of the year as ChatGPT, Bard, and Bing become better and add a robust ecosystem. Unless you‚Äôre literally in the business of selling LLMs, an LLM isn‚Äôt a product! It‚Äôs an engine for features.

By treating an LLM like an engine that creates a Honeycomb query, we shifted the focus of our work from being primarily about shipping an LLM interface to users and about extending our product UI. It may have been cheaper to create ‚ÄúHoneycombGPT‚Äù and ship a crappy version of ChatGPT with Honeycomb querying as its sole capability (sans prompt injection), but we felt that was uninspiring and the wrong interface for most people.¬†

The bulk of the work in building Query Assistant was no different from most product work: design and design validation, scoping things (in this case, very aggressively to meet a one¬† month deadline), making decisions based on roadblocks found in development, and a lot of dogfooding and as much external validation of the experience as possible. Don‚Äôt mistake an LLM for a product, and don‚Äôt think it can replace standard product work.

LLMs force you to address legal and compliance stuff

You can‚Äôt just plop some API calls to OpenAI into your product, ship to customers, and expect that to be okay if you have anything more than a small handful of customers. There are customers who are extremely privacy-minded and will not want their data, even if it‚Äôs just metadata, involved in a machine learning model. There are customers who are contractually obligated to be privacy-minded (such as customers handling healthcare data), and regardless of how they feel about LLMs, need to ensure that no such data is compromised. And there are customers who sign specific service agreements as a part of an enterprise deal.

Some things that we did:

  • Do a full security and compliance audit of LLM providers. Spoiler alert: only OpenAI could meet our requirements for now. Props to them for building a robust service!
  • Draft new terms and conditions that detail what data we send to OpenAI, that we make no claim over data sent or data received, and that we do not guarantee particular results/outcomes, etc.
  • Figure out what (if any) terms need to be changed in our overall terms of service, which hadn‚Äôt been updated since 2021
  • Ensure these terms and conditions are accessible within the UI itself
  • Ensure there are easy and obvious controls to turn the feature completely off
  • Flag out any customer who signs a BAA with us. Although OpenAI‚Äôs platform controls would likely satisfy each agreement, we‚Äôd need to work with each customer on a case by case basis, and we didn‚Äôt want to hold up our initial release

Under a longer timeline, none of these would have been special or challenging, but we had to do this in under a month alongside all the other product, engineering, and go-to-market work. You might think it‚Äôs unnecessary to do this sort of thing for an initial launch, but it is if you care about keeping your customers trusting and happy.
