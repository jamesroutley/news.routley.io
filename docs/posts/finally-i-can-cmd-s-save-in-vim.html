As a standards-interested person, I love catching up with the progress of people who are trying to improve other, seemingly-intractable problems in software. One such guy is Andy, the creator of OILS (and fellow recurser!), a backwards-compatible shell that aims to be an upgrade path from Bash to… you know, something better than Bash. I’ve learned a lot from his writing over the years, particularly about xargs.

Another related one is terminals.

Terminal Vim

Here’s a problem I’ve had for years: I would like to save my Vim buffer with Cmd+S.

The default way to save a Vim files to type :w then hit enter in normal mode. That’s fine, but it’s way too cumbersome for an action that I am going to do many, many times in a single editing session.

Vim supports keyboard shortcuts with modifier keys like this. <C-s> would be Ctrl+S, <D-s> for Cmd+S and so on. In theory, all I should have to do is add the following line to my .vimrc file.

nnoremap <D-s> :w<cr>

The nnoremap command maps one set of keystrokes in normal mode to another set of keystrokes. In this case, it tells Vim that whenever I type <D-s> (Cmd+S), Vim should instead act as :w and then enter (<cr>, it stands for “carriage return”).

I’ve been using iTerm2 on MacOS for years. Someone at my first job told to install it because it was better than the built-in terminal. This was in the “frontend and backend are separate repos” phase of web development, and pre-Docker, so there were a lot of services that had to be run at the same time in different terminals. I learned how to do split panes and that’s pretty much the only terminal feature I’ve ever needed.
